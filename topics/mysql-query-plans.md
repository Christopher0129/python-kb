# MySQL 查询计划专题

这个专题关注 MySQL 查询为什么慢、索引为什么没用上，以及如何从执行计划定位真正的瓶颈。

## 适用场景

- SQL 在测试环境没问题，上线后突然变慢
- 索引已经建了，但扫描行数依然很高
- 分页、排序、模糊查询和多表关联成本难以判断

## 常见主题

- `EXPLAIN` / `EXPLAIN ANALYZE`
- 覆盖索引与回表
- 复合索引顺序
- 排序、分页与临时表
- 慢查询日志与执行计划回归

## 设计要点

- 执行计划要结合真实过滤条件、数据分布和排序要求一起看，不能只看有没有走索引。
- 复合索引是否有效，取决于最左前缀、过滤选择性和排序列组合。
- 大偏移量分页通常不是“语法问题”，而是扫描范围与回表成本过高。
- SQL 优化要和业务访问模式一起改，不要只在单条语句上做局部修补。
