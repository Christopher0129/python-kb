# 观测埋点专题

这个专题关注代码里应该在哪里埋日志、指标和 trace，以及怎样避免埋点越多越混乱。

## 适用场景

- 服务出问题时只能看到“报错了”，看不到过程
- 请求链路很长，定位不到瓶颈阶段
- 观测数据很多，但无法支持真实排障

## 常见主题

- trace / span 埋点
- 业务指标埋点
- 请求和任务生命周期
- 异步上下文传播
- 埋点命名与标签规范

## 设计要点

- 观测埋点要围绕关键业务阶段，而不是机械地给每个函数都打点。
- 指标标签和日志字段必须提前规范，否则后期几乎无法治理。
- 异步任务、消息消费和后台 worker 需要单独处理上下文传播。
- 埋点设计要兼顾排障价值和采集成本，避免高基数字段直接进入指标。
